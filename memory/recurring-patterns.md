# Recurring Patterns — cross-deck learning

Cross-deck learning, so the editor compounds value across decks it has critiqued. Each entry is **one line**: a failure pattern that has recurred across multiple decks, tagged by domain. Over time this sharpens what the editor is alert to — without ever leaking one founder's work into another's session.

## Governance (read before touching this file)

- **Append on approval only.** The editor *proposes* an entry at the end of a pass (step 5 of `skills/critique-pass.md`); it appends **only on the founder's explicit yes.** Never write silently.
- **No deck content, ever.** Entries are abstract patterns, never quoted lines, numbers, names, or anything traceable to a specific founder's draft. One founder's deck must never surface in another's session. This is the privacy line — do not cross it.
- **One line each, domain-tagged.** Format: `[domain-tag] <the recurring pattern>`. Example shape: `[icm-ai] the hidden human — solution slides bury the human-in-the-loop as if it were a weakness`.
- **Domain tags scope relevance.** After a re-domain (swap `_config/`), entries whose tag doesn't match the active domain are **ignored**, not deleted — they stay for if that domain returns.
- **Capped at ~30 entries.** On overflow, prune the oldest, least-recently-seen entry. Patterns that keep recurring stay; one-offs age out.
- **A pattern is not a rule.** These entries prime what to look for; they never override the gates or license a generic note. A recurring pattern still has to be quoted and pass the paste test on *this* deck to become a note.
